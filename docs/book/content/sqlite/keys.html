Почему одного имени недостаточно?

У двух игроков вполне может быть одинаковое имя. Если написать WHERE name = 'Alex', база изменит обе подходящие строки. Нужна колонка, значение которой однозначно указывает ровно на одну запись.

Имена повторяются, идентификаторы — нет
idnamelevel
1Alex5
2Mira12
3Alex8

Первичный ключ

PRIMARY KEY назначает колонку главным идентификатором строки. Значения первичного ключа не повторяются и не бывают NULL. У таблицы может быть только один первичный ключ.

AUTOINCREMENT поручает SQLite самой выдавать следующий номер: 1, затем 2, затем 3. Колонку id больше не нужно указывать в INSERT.

Номер только что добавленной строки

Результат INSERT хранит созданный идентификатор в свойстве last_insert_id. Если команда ничего не добавляла, там будет null.

Используется to_int64(), потому что SQLite рассчитана и на таблицы с очень большим количеством строк.

Другие уникальные значения

Первичный ключ один, но запретить повторы можно и в других колонках. Например, отображаемые имена игроков могут совпадать, а логины — нет.

Допустимые строки: name повторяется, login остаётся уникальным
idnamelogin
1Alexalex_fox
2Alexalex_wolf

Попытка добавить ещё одну строку с логином alex_fox завершится ошибкой ограничения UNIQUE. База не станет молча портить правило, которое мы ей задали.
