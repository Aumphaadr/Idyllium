Ошибка выполнения останавливает программу

В прошлом уроке мы разделили ошибки на два вида. Ошибку компиляции Idyllium замечает ещё до запуска. А ошибка выполнения возникает, когда программа уже работает: например, пытается поделить число на ноль, прочитать отсутствующий файл или преобразовать слово в число.

Обычно после такой ошибки программа останавливается. Строки, расположенные ниже опасной операции, уже не выполняются.

main.idyl:5: runtime error: division by zero

Иногда остановка является правильным поведением: программа обнаружила серьёзную проблему и не знает, что делать дальше. Но иногда мы заранее понимаем, что может пойти не так, и умеем предложить запасной путь.

Опасный участок внутри try

Блок try содержит код, который мы хотим попытаться выполнить. Слово try так и переводится: «попробовать».

Если внутри этого блока возникает ошибка выполнения, Idyllium сразу переходит в следующий блок catch. Код внутри catch описывает запасное поведение.

На ноль делить нельзя Программа продолжает работу

Обратите внимание: строка с делением не завершилась, но вся программа уже не остановилась. После catch выполнение продолжилось обычным образом.

Как узнать, что именно произошло

Короткая форма catch { ... } подходит, когда для любых ошибок у нас один ответ. Если нужны подробности, между круглыми скобками можно создать имя для пойманной ошибки: catch (error).

Причина: division by zero Файл: main.idyl Строка: 6

У пойманной ошибки есть четыре полезные возможности:

  • error.message — короткая причина;
  • error.file — имя файла;
  • error.line — номер строки;
  • error.to_string() — полное сообщение одной строкой.

Имя error не является особым словом. Можно написать catch (problem) или выбрать другое понятное имя. Эта переменная существует только внутри своего блока catch, а её свойства предназначены только для чтения.

Запасное значение

Обработка ошибки не обязана ограничиваться сообщением. Например, функция может попытаться преобразовать строку в число, а при неудаче вернуть заранее выбранное запасное значение.

42 0 10

Такой приём полезен, только если запасное значение действительно имеет смысл. Нельзя оборачивать половину программы в один огромный try и делать вид, что любой сбой неважен. Чем меньше опасный участок, тем понятнее, какую проблему мы собирались обработать.

finally выполняется в любом случае

После try и catch можно добавить блок finally. Он выполняется и после успеха, и после пойманной ошибки. Это удобное место для обязательного завершающего действия: закрытия файла, отмены незавершённой транзакции или остановки ресурса.

Результат: 4 Попытка завершена Деление не удалось Попытка завершена

У конструкции обязательно должен быть хотя бы один блок: либо catch, либо finally. Допустимы обе формы: try + catch и try + finally. Чаще всего для понятной обработки используются сразу все три блока.

Что try поймать не может

Конструкция try работает только с ошибками выполнения Idyllium. Ошибку компиляции поймать нельзя: программа с неверным синтаксисом или несовместимыми типами вообще не будет запущена. Остановка программы пользователем тоже не превращается в обычную ошибку и не отменяется через catch.

Главная идея проста: try не прячет проблемы, а позволяет заранее описать разумный план на случай ожидаемой неудачи.

Попробуйте сами

  1. Попросите пользователя ввести целое число и выведите понятную подсказку, если вместо числа введён текст.
  2. Попытайтесь открыть отсутствующий файл и обработайте ошибку без остановки программы.
  3. Добавьте finally, который сообщает о завершении каждой попытки.
