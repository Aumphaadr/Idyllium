Зачем ещё один способ хранить данные?

В разделе про JSON мы уже сохраняли игровые данные в файлы. JSON прекрасно подходит, когда программе удобно прочитать документ целиком: например, настройки или одно сохранение. Но представим сервер игры с десятью тысячами игроков. Чтобы найти одного игрока, не хочется каждый раз перебирать огромный массив и заново сохранять весь файл после изменения одного уровня.

База данных хранит множество записей и умеет искать, добавлять и изменять нужные строки. SQLite особенно удобна для знакомства с базами: ей не нужен отдельный сервер. Вся база помещается в один файл рядом с проектом.

Таблица, строки и колонки

Таблица похожа на знакомую электронную таблицу. В таблице players одна строка описывает одного игрока, а каждая колонка хранит один вид данных.

idnamelevelnickname
1Liam7NULL
2Mira12Star

Название таблицы и названия колонок придумывает автор программы. Значение NULL пока можно читать как «данные неизвестны». Мы спокойно разберём его в отдельном уроке.

Открываем базу

Библиотека называется sqlite. Функция sqlite.open() открывает существующий файл базы или создаёт новый, если такого файла ещё нет.

Путь: game.db База открыта: true

Переменная db хранит открытое соединение с базой. Метод close() завершает работу с ним. После закрытия свойство is_open станет равным false.

Первая таблица

Команды для базы пишутся на языке SQL и передаются в метод execute(). Команда CREATE TABLE создаёт таблицу.

После CREATE TABLE написано придуманное нами имя таблицы — players. В круглых скобках через запятую перечисляются колонки. У каждой колонки сначала записано имя, затем тип данных: INTEGER означает целое число, а TEXT — текст.

Такой запрос предназначен для пустой базы: при повторной попытке создать уже существующую таблицу SQLite честно сообщит об ошибке. Позже мы научим запрос спокойно переживать повторный запуск, а пока перед новым экспериментом можно удалить созданный файл базы.
