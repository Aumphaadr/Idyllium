# Documentation Migration Plan

Цель: перенести старую документацию Idyllium в IdylliumNext так, чтобы сохранить авторский педагогический стиль, но привести синтаксис, примеры и сайт к актуальному состоянию языка.

## Главный принцип

Старая документация — это не просто справочник. Это живой детский учебник: объяснения, примеры, предупреждения, типичные ошибки, аккуратное ведение ученика от простого к сложному.

При миграции нельзя превращать её в сухой API reference. Сухая документация
существует отдельно и не заменяет учебник.

## Учебная линия

Новый порядок крупных разделов:

1. Консоль.
2. Виджеты.
3. ООП.
4. Canvas.
5. JSON.
6. Следующие разделы по мере появления языка и библиотек.

Canvas должен идти после ООП, потому что осмысленные Canvas-проекты естественно используют классы, объекты, callbacks и разделение кода по файлам.

## Что переносить автоматически

Первый мигратор должен без умных языковых правок перенести:

- структуру разделов из старого `lessons.json`;
- HTML-текст уроков;
- заголовки, подзаголовки, списки, поясняющие блоки;
- `idyl-code-block`;
- `idyl-output-block`;
- навигацию между уроками;
- локальные стили и демонстрационные GUI-блоки, если они не ломают новый сайт.

Мигратор не должен пытаться автоматически "исправить" язык там, где нужен педагогический выбор.

## Что править вручную после миграции

### Консоль

- Добавить урок или фрагмент про `console.clear()`.
- Показать пример:

```idyllium
use console;

main() {
    console.writeln("нар");
    console.clear();
    console.writeln("котики");
}
```

Ожидаемый вывод:

```text
котики
```

Педагогическая мысль: первая строка была выведена, но затем консоль очистили; осталось только то, что вывели после очистки.

### Цвета

- Добавить `colors` в один из ранних уроков, где это не перегружает ученика.
- В разделе "Виджеты" сделать отдельный урок о цветах виджетов.
- Показать `colors.RGB`, `colors.RGBA`, `colors.HEX`, `colors.HSL`, константы `colors.WHITE`, `colors.BLACK`, `colors.RED`, `colors.GREEN`, `colors.BLUE`, `colors.TRANSPARENT`.
- Показать свойства `text_color`, `background_color`, `border_color`.
- Показать наследование `text_color` и `background_color` от окна к потомкам и от `Frame` к вложенным виджетам.
- Цветовые свойства принимают `colors.Color`; строковый HEX нужно явно
  преобразовать через `colors.HEX()`.

### Виджеты

- Проверить урок `ProgressBar`.
- Решить, добавляем ли `text_color` и `background_color` для `gui.ProgressBar`.
- Если добавляем, обновить урок цветов так, чтобы `ProgressBar` тоже участвовал в общей модели цветовых свойств.

### Canvas

- Создать новый раздел после ООП.
- Использовать текущий draft из `spec/some_canvas_game` как источник направления.
- Показать Canvas как обычный `gui`-виджет.
- Показать `drawable`-объекты, callbacks, события клавиатуры/мыши, игровой цикл через `on_update`.
- Показать связь Canvas с `colors`.

### Ошибки

- Сохранять отдельные уроки с типичными ошибками.
- Оставлять вкрапления ошибочного кода внутри обычных уроков, когда ошибка нужна именно в контексте.
- Для каждого teaching error по возможности показывать ожидаемый текст ошибки.
- Ошибки должны быть короткими, читаемыми и пригодными для перевода.

### Запрещённые или устаревшие темы

- Не добавлять операторы `++` и `--`.
- Старые уроки про инкремент/декремент нужно переписать под `+= 1` и `-= 1`, либо переосмыслить тему целиком.

## Технический план

### Шаг 1. Inventory

Скрипт собирает список старых уроков:

- путь;
- раздел;
- номер урока;
- title;
- subtitle;
- code blocks;
- output blocks;
- локальные стили;
- ссылки на предыдущий/следующий урок.

Результат можно сохранить в JSON, чтобы видеть объём работ.

Текущий инструмент:

```text
npm run docs:inventory
```

Он создаёт:

- `docs/migration/lesson-inventory.json`;
- `docs/migration/outdated-syntax-report.md`.

### Шаг 2. Raw Migration

Скрипт переносит старые уроки в новый каталог документации без смысловых правок.

Требования:

- не менять текст объяснений;
- не менять код автоматически, кроме механической нормализации путей и обёрток;
- сохранить возможность ручной правки каждого урока;
- сгенерировать новый индекс разделов.

Текущий инструмент:

```text
npm run docs:migrate
```

Он создаёт сырой перенос в `packages/docs`.

Важно: это не финальная документация, а бережный слепок старого живого учебника. Скрипт специально не исправляет код примеров и не переписывает объяснения, чтобы не потерять педагогический стиль.

### Шаг 3. Outdated Syntax Report

Отдельный отчёт помечает уроки, где вероятны ручные правки:

- `++` / `--`;
- удалённое строковое свойство цвета у `Label`;
- HEX-строки вместо `colors`;
- Canvas-черновики или отсутствующие Canvas-уроки;
- код без `main()`;
- teaching errors;
- места, где ожидаемый вывод не соответствует новому runtime.

### Шаг 4. Manual Rewrite

Ручная редактура:

- обновить синтаксис;
- сохранить авторский стиль объяснений;
- добавить новые примеры;
- добавить ожидаемый output и expected errors;
- проверить код компилятором.

### Шаг 5. Static Site

Собрать IDE, учебник и документацию как единый статический сайт для GitHub
Pages.

Функции:

- навигация по разделам;
- подсветка синтаксиса;
- копирование кода;
- показ expected output;
- показ expected error;
- ссылка на Web IDE в шапке, но без кнопки "открыть этот пример в IDE".

Текущий инструмент:

```text
npm run docs:site
```

Он создаёт GitHub Pages-слой прямо в `docs/`:

- `docs/index.html` и корневые ассеты — Web IDE по адресу `/`;
- `docs/book/` — SPA учебника, manifest и HTML-фрагменты уроков;
- `docs/reference/` — SPA взрослой документации и `api.json`, собранный из
  standard library registry;
- `docs/ide/` — перенаправление со старого адреса IDE на корень сайта.

URL уроков имеют hash-форму вроде `/book/#/console/hello`, а страницы API —
`/reference/#/image/Static`. Такой routing не требует настройки сервера и
стабильно работает на GitHub Pages. Нумерация уроков вычисляется по порядку в
`docs/book/lessons.json`, поэтому файлы уроков называются смысловыми slug-ами,
а не `001_...`, `002_...`.

Исходники оболочек находятся в `packages/docs-book/` и
`packages/docs-reference/`. Педагогический текст остаётся в
`docs/manual-content/`; ручная правка сгенерированных файлов внутри
`docs/book/` и `docs/reference/` не сохраняется при следующей сборке.
