# IdylliumNext Backlog

Живой список задач и идей, которые важны для проекта, но не обязаны выполняться
прямо сейчас.

## Правила

- Синтаксис Idyllium не менять без отдельного обсуждения.
- Папка `spec/lessons` и draft-примеры остаются источником языковых ожиданий.
- Отложенная идея не считается принятой спецификацией, пока по ней нет понятного синтаксиса, тестов и примеров.
- Документация может показывать код, вывод и кнопку копирования, но не должна отправлять примеры напрямую в IDE.
- Для детской документации один пример кода не считается полноценным уроком: урок должен постепенно ставить проблему, объяснять мотивацию, давать несколько малых шагов/примеров, показывать вывод и только затем приводить итоговый код.
- Операторы `++` и `--` в Idyllium не добавляем.
- Пока у проекта нет заметной базы длительно поддерживаемых пользовательских
  программ, активное легаси не сохраняем: после утверждения замены старый API
  удаляется, а спека, тесты, учебник и документация переводятся на новый.

### Функции и вызовы

Статус: `typed main`, дефолтные аргументы и базовые именованные аргументы реализованы; следующая своевременная задача - UX-доводка подсказок и spec/docs.

Что уже есть:

- педагогическое сокращение `main() { ... }`;
- обычные функции вида `int function sum(int a, int b)`;
- методы классов, статические методы, конструкторы;
- inline callback-функции вида `button.on_click = void function() { ... };`;
- присваивание именованных функций в callback-свойства;
- `main` в форме `main() { ... }` и в форме `T function main() { ... }`;
- значения по умолчанию для параметров функций, методов, конструкторов и inline callback-функций;
- именованные аргументы для функций, методов, конструкторов, stdlib-функций и user modules;
- рекурсия;
- readable compile/runtime errors для многих типичных ситуаций.

Важная техническая деталь:

- `math.floor()`, `math.ceil()`, `math.round()`, `json.Value()` и `to_pretty_json()` уже умеют принимать разное количество аргументов, но это не языковые дефолтные аргументы. Сейчас это реализовано через `minArguments` и ручную runtime-логику стандартной библиотеки.
- Пользовательские функции, методы, конструкторы и inline callback-функции уже имеют значения по умолчанию.
- AST вызова хранит аргументы как пару `name/value`, где `name` может отсутствовать.
- `FunctionType` хранит типы и минимальное количество аргументов; имена параметров для прямых вызовов берутся из деклараций, stdlib registry и user module metadata.

Именованные аргументы:

- Реализованы именованные аргументы:
  `sub(arg1=50, arg2=30)`.
- Реализована смена порядка именованных аргументов:
  `sub(arg2=50, arg1=30)`.
- Смешивание позиционных и именованных аргументов разрешено только в порядке:
  сначала позиционные, потом именованные.
- Позиционный аргумент после именованного запрещён:
  `sub(arg1=50, 30)` -> compile error.
- Повторная передача одного параметра запрещена:
  `sub(50, arg1=30)` -> compile error.
- Неизвестное имя аргумента запрещено:
  `sub(left=50)` -> compile error, если у функции нет параметра `left`.
- Для variadic-функций вроде `console.write(...)` именованные аргументы на первой итерации запрещены.
- Для callback-вызовов и значений типа `function(...)` именованные аргументы поддерживаются только там, где компилятор точно знает имена параметров. Если известны только типы, вызов остаётся позиционным.

Значения по умолчанию:

- Реализованы объявления вида:
  `void function print_num(int num = 0)`.
- Реализованы для функций, методов, конструкторов, inline callback-функций и user modules.
- Дефолтное значение проверяется так же строго, как обычное присваивание параметру.
- Дефолтное значение может использовать предыдущие параметры:
  `int function add_twice(int first, int second = first)`.
- Параметры со значением по умолчанию идут после обязательных параметров:
  `void function f(int a, int b = 0, int c = 0)` - можно;
  `void function f(int a = 0, int b)` - compile error.

Особый случай `main`:

- `main() { ... }` остаётся педагогическим сокращением и считается `void function main()`.
- Обычная форма после изучения функций уже разрешена:
  `int function main() { return 0; }`.
- В программе может быть не больше одной функции `main`: нельзя одновременно иметь `main() {}` и `int function main() {}`.
- `main` не должен принимать параметры.
- Возвращённое значение пока вычисляется и игнорируется Web IDE/GUI preview;
  отдельное решение о превращении `int` в exit code CLI/VSIX можно принять позже.
- `main` может иметь любой возвращаемый тип, доступный обычным функциям: `void`, `int`, `float`, `string`, `bool`, `char`, массивы, классы и типы библиотек. Если тип не `void`, компилятор должен требовать `return` подходящего значения.

Оставшаяся UX-доводка:

- Language service обновлён: completion имён аргументов внутри вызова работает в ядре, Web IDE и VSIX.
- Signature help уточнён для named args и default values; Web IDE получила Monaco signature help provider.
- Добавить spec-примеры и documentation snippets: успешные вызовы, ошибки, методы, конструкторы, callbacks, typed `main`.

### Битовые сдвиги в `types`

Статус: реализовано для всех целых и дробных типов библиотеки.

- `shift_left(bits)` и `shift_right(bits)` работают с битовой ячейкой
  фиксированной ширины;
- крайние биты молча теряются, новые позиции заполняются нулями;
- правый сдвиг всегда логический и не сохраняет знак;
- сдвиг на ширину типа или больше даёт нулевую ячейку;
- у `float32` и `float64` сдвигается IEEE-представление, после чего биты снова
  интерпретируются как дробное число;
- результат сохраняет исходный `types.*`-тип и поддерживает цепочки методов;
- отрицательное количество меняет направление: `shift_left(-N)` равен
  `shift_right(N)`, а `shift_right(-N)` равен `shift_left(N)`;
- приёмочные примеры лежат в `spec/some_types`, frozen-пример - в
  `spec/lessons/examples/cli/020_types/011.idyl`.

  `rename` и безопасный `remove` с явным флагом `recursive`.
- Мутации Node/VSIX runtime ограничены корнем проекта, не проходят через
  символические ссылки наружу и не перезаписывают существующую цель.
- Web IDE синхронизирует не только созданные файлы, но также каталоги,
  удаления и переименования через runtime tombstones.
- У `file.istream` и `file.ostream` реализовано readonly-свойство `is_open`.

### Семантическая подсветка

- Semantic tokens формируются компилятором и используются Web IDE и VSIX.
- Классы, пространства имён, функции, методы, свойства, параметры и переменные
  различаются по фактическому смыслу, включая пользовательские модули и классы.
- Переменная с заглавной буквы больше не становится классом из-за формы имени.
- При добавлении новых синтаксических сущностей расширять общий semantic API,
  а не создавать отдельные regex-эвристики в редакторах.

### Именованные константы

Статус: реализовано вертикально в compiler, runtime/codegen, semantic tooling,
Web IDE, VSIX grammar, spec-примерах, учебнике и справочнике.

- синтаксис: `const int N = 240;`;
- поддержаны локальные и верхнеуровневые объявления;
- инициализатор обязателен;

### Визуальный конструктор GUI

Статус: идея отложена, требуется отдельное проектирование.

Хотелка: визуальный редактор, который помогает расставлять виджеты и тем самым

Ключевое ограничение: разработка идёт итеративно. Ученик не всегда сначала
создаёт все виджеты, а потом пишет код; часто он добавляет часть интерфейса,
пишет обработчики, потом добавляет новые виджеты и снова пишет код.

Возможные направления:

- Генерировать `app.idyl` целиком. Риск: при повторной генерации легко потерять
  написанные вручную функции, обработчики и пользовательский код.
- Генерировать XML/JSON/UI-файл и подключать его из `Application`. Плюс: код
  приложения живёт отдельно от разметки, визуальный редактор меняет только UI.
- Генерировать `app.idyl` с защищёнными managed-регионами, например
  `// <idyllium-ui-managed>` ... `// </idyllium-ui-managed>`. Риск: детям и
  внешним редакторам легко случайно испортить служебные маркеры.
- Генерировать отдельный файл `app_ui.idyl` с функцией сборки интерфейса, а
  пользовательский `app.idyl` хранит логику. Нужно продумать, как удобно
  связывать поля виджетов и обработчики.

Предварительное направление: изучить XML/JSON/UI-файл как источник интерфейса,
а в `app.idyl` оставить класс `Application`, состояние и методы. Это лучше
переживает итеративную разработку.

### Canvas: геометрия, трансформации и origin

Статус: базовая модель реализована вертикально в runtime, registry и общем
renderer; ручные сценарии лежат в `spec/some_origins/manual_tests`.

Реализовано:

- `x/y` задают мировую позицию origin, `set_origin(x, y)` — его локальную точку;
- позиции всех drawable-объектов имеют тип `float`, поэтому результаты
  `sin()`/`cos()` можно присваивать координатам без округления;
- origin по умолчанию равен `(0, 0)`, положительный угол вращает по часовой
  стрелке;
- `contains(x, y)` учитывает origin, rotation и scale;
- `intersects(drawable.Drawable)` работает симметрично через общие внутренние
  формы: circle, convex polygon и capsule для толстой Line;
- касание считается пересечением, оформление и прозрачные пиксели не меняют
  collision geometry;
- Circle использует локальный квадрат `2r × 2r`; `set_origin(r, r)` делает
  `x/y` координатами центра;
- Sprite учитывает естественные размеры картинки, rotation и отрицательный
  scale; Line моделируется капсулой с круглыми концами.
- Text измеряет TTF, OTF, WOFF и WOFF2 через общий
  `RuntimeFontMetricsService`; Canvas-renderer использует те же advance-метрики
  без kerning;
- Text без явно назначенного Font использует поставляемый Source Code Pro и в
  renderer, и при расчёте `contains()` / `intersects()`.
- в учебник добавлен урок про origin, вращение и движение по окружности с прямой
  связью между `rotation`, `sin()` и `cos()`.

Уроки про origin и готовые `contains()` / `intersects()` добавлены после ручного
вывода формул; отдельный ручной сценарий проверяет повёрнутый Text.

### Библиотека `sqlite` / локальная база данных

Статус: первая версия runtime, frozen spec и учебная линия реализованы; нужна
ручная педагогическая редактура уроков после проверки автором.

Уже есть:

- единый `Database.execute()` / `Statement.execute()` и буферизованный
  `sqlite.Result`;
- строгие именованные параметры `:name`, универсальный `bind()` и
  типизированные `bind_*()` для явного SQL storage class;
- `sqlite.Value`, языковой `null` и точные SQLite INTEGER через `types.int64`;
- явные `begin_transaction()`, `commit()` и `rollback()`;
- `exec_script()`, создание и сохранение бинарных `.db`-файлов;
- одна реализация на `sql.js` для CLI, VSIX и Web IDE без CDN;
- 13 frozen-программ: 10 рабочих сценариев и 3 ожидаемые runtime error;
- раздел учебника из 14 уроков: от первой таблицы до транзакций, скриптов,
  статистики, ключей, связей между таблицами и типичных ошибок;
- отдельные SQL-блоки с подсветкой и таблицы, показывающие исходные данные и
  результаты запросов.
- приёмочные тесты всех программ из `05_recommended_api`.

Важная деталь `bind()`:

- `bind()` принимает обычные числа, строки, символы, bool, `null`, типы из
  `types` и `sqlite.Value`;
- `bind_float("value", 2.0)` остаётся полезным, потому что во время выполнения
  значения `2` и `2.0` неразличимы, а этот метод гарантирует SQL REAL;
- строгие методы не являются обязательной церемонией и не удаляются: это
  осознанный escape hatch для точного типа хранения.

Файлы концептов лежат в `spec/some_sqlite`:

- `01_cursor_api` - простой `Database`/`Result` API;
- `02_prepared_statement_api` - безопасные prepared statements с `bind_*`;
- `03_json_bridge_api` - получение результата SQL как `json.Array`;
- `04_table_record_api` - возможный детский CRUD-слой без явного SQL;
- `05_recommended_api` - собранный кандидат первой версии с `Database`,
  `Statement`, `Result`, `sqlite.Value`, именованными параметрами `:name`,
  транзакциями и SQL-скриптами.

Следующие отдельные задачи: ручная редактура уроков, дополнительное тестирование
в Web IDE/VSIX и затем возможный JSON-мост. `Table`/`Record` оставить на потом,
чтобы не прятать SQL слишком рано.
